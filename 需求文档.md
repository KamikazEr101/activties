



一、系统简介
高校学生活动全流程管理系统是一款为高校学生和活动管理员提供学生活动信息的发布、报名、管理等全流程服务的Web应用，主要为活动管理者提供活动的创建、查询、发布、编辑、删除等功能，为活动参与者提供报名功能
通过本系统，期望能够显著提升活动举办方对各类学生活动的管理水平、参与者对各类学生活动信息的获取能力
二、需求分析
2.1用户角色：
用户	特征
参与学生	1.	非注册用户： 无需注册，通过外部链接即可访问活动信息并报名。
2.	便捷性需求： 期望通过简单、快速的方式完成报名和签到。
3.	信息输入： 需要提供个人报名信息（姓名、手机号、学院）。
4.	反馈需求： 关注报名结果的及时反馈。
5.	关注点： 活动详情、报名流程、报名结果、签到结果。
活动管理员	1.	权限性： 作为系统正式用户，拥有创建、编辑、删除、发布活动等最高管理权限。
2.	负责任： 对活动信息的准确性、完整性和及时性负责。
3.	操作频率： 相较于参与学生，操作频率可能较低，但每次操作的重要性高，需精细管理活动全生命周期。
4.	关注点： 活动发布效果、报名情况、活动执行进展和数据分析。

4.2. 控制器将数据模型转发给前端页面，并渲染活动详情页面
5.	前端展示： 网页浏览器向参与学生展示加载完成的活动详情内容，包括文本描述、图片、报名按钮等
6.	成功查看： 参与学生成功浏览活动详情
扩展流程	1.	活动ID无效或活动不存在：
1.1. 系统在数据库中未找到对应的活动ID
1.2. 系统返回对应状态码及错误信息，并在前端页面显示“您访问的活动不存在或已下架”的提示信息

（2）报名活动
ID	2	名称	报名活动
创建日期	2025.10.20	最后更新日期	2025.10.20
参与者	参与学生
触发条件	1.	参与学生在“查看活动详情”页面点击“立即报名”按钮
前置条件	1.	参与学生已成功查看活动详情
2.	活动处于可报名状态，且未截止报名，名额未满
后置条件	1.	参与学生成功提交报名信息，系统记录报名数据
2.	系统向参与学生显示报名成功提示。若失败则提示“报名已截止”、“填写信息错误”、“名额已满”或“请勿重复报名”
3.	活动表的报名人数统计信息得到更新
流程	1.	点击报名按钮： 参与学生在活动详情页点击“立即报名”按钮
2.	显示报名表单： 系统跳转到报名表单页面，页面显示活动名称，并要求参与学生填写个人报名信息（姓名、手机号、学院）
3.	填写并提交信息： 参与学生填写所需信息后，点击“提交报名”按钮
4.	系统接收请求： Spring Boot接收到提交报名信息的HTTP POST请求，请求体中包含活动ID和参与学生的个人信息
5.	服务层处理与校验：
5.1. 控制器调用报名服务层Service
5.2. 服务层首先执行各种校验：
活动状态校验： 验证活动是否仍在报名期、是否已下架
名额校验： 从MySQL检查活动剩余名额
重复报名校验： 根据活动ID和学生手机号校验是否已报名
信息合法性校验： 校验姓名、手机号（格式）、学院等信息的完整性和合法性
5.3. 若所有校验通过，服务层将报名信息写入MySQL数据库 5.4. 服务层通知更新MySQL中的活动报名人数统计
6.	响应报名结果：
6.1. 控制器将报名结果（成功/失败及原因）返回给前端
7.	前端显示结果： 网页浏览器向参与学生显示报名成功或失败的提示信息
8.	报名成功/结束： 参与学生完成报名流程
扩展流程	1.	已过报名截止时间：
1.1. 系统校验发现活动已过报名截止时间
1.2. 系统返回对应状态码及错误信息，并在前端显示“抱歉，该活动报名已截止”的提示信息
2.	报名名额已满：
2.1. 系统校验发现活动报名名额已满
2.2. 系统返回对应状态码及错误信息，并在前端显示“抱歉，该活动名额已满”的提示信息
3.	重复报名：
3.1. 系统校验发现该参与学生已报名过此活动
3.2. 系统返回对应状态码及错误信息，并在前端显示“您已报名过此活动，请勿重复报名”的提示信息
4.	报名信息不完整或不合法：
4.1. 系统校验发现参与学生提交的姓名、手机号、学院等信息有误或不完整
4.2. 系统返回对应状态码及错误信息，并在前端表单旁显示具体错误提示（如“手机号格式不正确”、“学院不能为空”）
4.3. 参与学生修正信息后可重新提交

ID	3	名称	活动签到
创建日期	2025.10.20	最后更新日期	2025.10.20
参与者	参与学生
触发条件	1.	参与学生在活动现场扫描签到二维码
2.	参与学生在活动详情页面点击“签到”按钮
前置条件	1.	参与学生已成功报名该活动
2.	活动处于签到时间段内（活动开始前30分钟至活动结束后1小时）
3.	参与学生可以访问互联网
后置条件	1.	系统记录参与学生的签到时间和状态
2.	参与学生的签到状态更新为“已签到”
3.	系统向参与学生显示签到成功提示
流程	1.	发起签到请求：参与学生通过扫描二维码或在活动详情页点击“签到”按钮，向系统发起签到请求
2.	系统接收请求：Spring Boot接收到签到HTTP POST请求，携带活动ID和学生手机号
3.	身份验证：
3.1. 控制器调用签到服务层Service
3.2. 服务层验证学生是否已报名该活动
3.3. 验证活动是否处于可签到时间段
4.	签到处理：
4.1. 所有验证通过后，服务层更新MySQL中该学生的签到状态为“已签到”
4.2. 记录签到时间戳
5.	响应结果：控制器将签到结果返回给前端
6.	前端展示：网页浏览器向参与学生显示签到成功提示
7.	完成签到：参与学生完成签到流程
扩展流程	1.	未报名签到：
1.1  系统验证发现该学生未报名此活动
1.2. 系统返回错误信息，前端显示“您未报名此活动，无法签到”
2.	重复签到：
2.1. 系统验证发现该学生已签到
2.2. 系统返回错误信息，前端显示“您已完成签到，请勿重复操作”
3.	不在签到时间段：
3.1. 系统验证发现当前时间不在活动签到时间段内
3.2. 系统返回错误信息，前端显示“当前不在签到时间段内”

（5）创建活动
创建日期	2025.10.20	最后更新日期	2025.10.20
参与者	活动管理员
触发条件	活动管理员在管理后台点击“创建活动”按钮。
前置条件	1.	活动管理员已成功登录系统
2.	活动管理员拥有创建活动的权限
3.	活动基本信息已准备完整
后置条件	1.	系统成功创建新活动并保存到数据库
2.	活动初始状态设置为“未发布”
3.	活动管理员可以继续编辑或发布该活动
4.	系统记录活动创建日志
流程	1.	进入创建页面：活动管理员进入活动管理页面，点击“创建活动”按钮，系统跳转到活动创建表单页面
2.	填写活动信息：活动管理员填写活动基本信息：
活动名称（必填）
活动描述（必填）
活动开始时间（必填）
活动结束时间（必填）
活动地点（必填）
负责人信息（必填）
联系方式（必填）
报名截止时间
最大报名人数
活动海报/图片
3.	提交表单：点击“提交”按钮提交表单
4.	接收请求：Spring Boot接收到创建活动的HTTP POST请求，包含完整的活动信息
5.	数据校验：
5.1. 控制器调用活动服务层Service
5.2. 服务层校验活动信息的完整性和逻辑正确性
6.	数据存储：
6.1. 校验通过后，服务层将活动信息写入MySQL数据库
6.2. 活动状态设置为“未发布”
6.3. 记录创建管理员ID和时间戳
7.	文件处理：如果上传了活动海报，将文件保存到Minio/OSS对象存储
8.	记录日志：在MySQL中记录活动创建操作日志
9.	返回结果：控制器将创建结果返回给前端
10.	页面跳转：网页浏览器向活动管理员显示创建成功提示，并跳转到活动编辑页面
11.	完成创建：活动管理员可以继续完善活动信息或直接发布
扩展流程	1.	必填字段为空：
1.1. 系统校验发现必填字段为空
1.2. 系统返回错误信息，前端在相应字段旁显示“该字段不能为空”提示
2.	时间逻辑错误：
2.1. 系统校验发现报名截止时间晚于活动开始时间
2.2. 系统返回错误信息，前端显示“报名截止时间不能晚于活动开始时间”
3.	文件上传失败：
3.1. 活动海报文件上传到将/OSS失败
3.2. 系统记录错误日志，但仍完成活动创建，前端显示“活动创建成功，但海报上传失败”
4.	数据库保存失败：
4.1. Mybatis执行数据库操作失败
4.2. 系统回滚事务，前端显示“创建失败，请稍后重试”

流程	1.	触发查看操作：活动管理员在活动管理列表中找到目标活动，点击"查看报名"按钮
2.	接收请求：Spring Boot接收到查看报名列表的HTTP GET请求，携带活动ID参数
3.	缓存检查：控制器调用报名服务层Service，服务层首先检查Redis中是否存在该活动的报名列表缓存
4.	缓存命中处理：如果缓存命中，直接从Redis返回数据

9.	前端渲染：网页浏览器向活动管理员展示报名列表页面，包含搜索、筛选、分页、导出等功能
10.	完成查看：活动管理员成功浏览报名学生信息
扩展流程	1.	活动不存在处理：系统在数据库中未找到对应的活动ID，返回错误信息，前端显示"活动不存在"

3.	数据筛选处理：活动管理员在报名列表页面使用搜索或筛选功能，系统根据筛选条件重新查询MySQL数据库（不适用缓存）。
4.	缓存异常处理：Redis服务不可用，系统降级直接查询MySQL数据库，保证功能可用性
